Эта страница описывает payout-интеграцию в `simple-pay`: как собрать реквизиты, какие статусы проходят выплаты и какие дополнительные проверки нужны до production.

Для готовых примеров запросов и ответов используйте страницу [PAYOUT: способы и API-примеры](/doc/payout/02-integration/).

## 1. Базовая схема payout

В `payout` вы не выдаёте клиенту реквизиты для оплаты, а создаёте заявку на выплату средств клиенту.

Типовой поток выглядит так:

1. проверить доступные payout methods;
2. собрать `customer.requisites` в формате выбранного метода;
3. создать payout-ордер;
4. ждать callback или читать статус через `GET`;
5. завершить бизнес-операцию только после финального статуса.

## 2. Что подготовить заранее

### Ключи

- `Shop API key` для работы с payout-ордерами.
- `Balance API key`, если вы хотите заранее читать баланс магазина.
- `Signature key` для проверки callback.

### Что должно быть настроено у магазина

- доступные payout trade methods;
- достаточный баланс магазина;
- `callbackUrl`, если вы хотите получать асинхронные статусы.

Если баланс магазина ниже нужного, создание payout-ордера завершится бизнес-ошибкой `S10001`.

## 3. Пошаговый flow

### Шаг 1. Получить доступные payout methods

Вызовите:

- `GET /shop/trade-methods/payout`

Это источник истины для:

- списка доступных `paymentType`;
- необходимости указывать `bank`;
- набора обязательных полей в `customer.requisites`.

Важно: не хардкодьте обязательные поля на клиенте. Список полей зависит от конкретного trade method.

### Шаг 2. Собрать реквизиты получателя

Заполните `customer.requisites` по полям из trade method.

Примеры:

- для `sbp` обычно нужен `phone`;
- для `card2card` нужен `cardInfo`, а `cardholder` обычно опционален;
- для банковских методов могут требоваться `accountNumber`, `bic`, `swiftBic`, `beneficiaryName`, `taxId`.

### Шаг 3. Создать payout-ордер

Вызовите:

- `POST /shop/payout-orders`

После успешного создания ордер обычно не финализируется мгновенно. Нормальный первый рабочий статус после создания это `requisites`.

Это означает, что система приняла заявку и передала её в обработку.

### Шаг 4. Дождаться статуса

Рекомендуемый вариант:

1. принять callback на `integration.callbackUrl`;
2. проверить `signature`;
3. зафиксировать новый статус.

Резервный вариант:

- `GET /shop/payout-orders/{id}`
- `GET /shop/payout-orders/external/{externalOrderId}`

### Шаг 5. Обработать финальные статусы

Финальными для мерчанта обычно являются:

- `completed`
- `cancelled`
- `dispute`
- `error`

Статус `rejected` не стоит считать финалом автоматически. Это промежуточный сигнал, что текущая попытка обработки не удалась и выплата может быть переотправлена дальше по внутреннему процессу.

## 4. Статусы PAYOUT

| Статус | Что означает |
| --- | --- |
| `new` | ордер только создан |
| `requisites` | заявка принята, система подбирает/передаёт реквизитный сценарий в работу |
| `trader_payment` | выплата в процессе исполнения |
| `rejected` | текущая попытка исполнения отклонена, возможна повторная маршрутизация |
| `completed` | выплата успешно завершена |
| `cancelled` | выплата отменена |
| `dispute` | спорный кейс, нужен разбор |
| `error` | техническая ошибка, нужен контроль статуса |

### Частые `statusDetails`

- `shop` , выплата отменена магазином;
- `requisites_timeout` , не удалось обработать реквизитный сценарий;
- `max_rejects_exceeded` , превышен лимит повторных отклонений;
- `invalid_requisites` , реквизиты получателя некорректны;
- `payment_failed` , выплата не прошла;
- `payment_timeout` , истекло время исполнения.

## 5. Что особенно важно для payout

### Баланс магазина

Payout использует баланс магазина, поэтому перед массовым запуском полезно проверять:

- `GET /shop/assets`

Это особенно важно для сервисов, где выплаты создаются пакетно.

### `externalOrderId`

Технически поле опционально, но в production его лучше считать обязательным. Оно помогает:

- безопасно переживать таймауты;
- исключать дубли;
- дочитывать payout по вашей внутренней операции.

### Автоподстановка банка для `card2card`

Если payout создаётся с `payment.type = card2card` и `payment.bank` не передан, сервис пытается определить банк по BIN карты получателя. Если банк не распознан, система подставляет технический fallback-банк.

Практически это означает:

- для `card2card` банк можно не передавать;
- но лучше читать итоговый `payment.bank` в ответе и логировать его.

## 6. Callback payout

Механика такая же, как в `payin`:

- метод задаётся в `integration.callbackMethod`;
- статусы передаются в query string;
- при `post` тело запроса пустое;
- callback может быть отправлен повторно;
- статус доставки отражается в `integration.callbackUrlStatus`.

Типовой callback:

```text
[[PAYOUT_CALLBACK_URL]]?id=b4ad11f1-10b3-4684-8fe4-2d6f3969e77a&amount=1200&customerId=payout-10001&status=completed&externalOrderId=merchant-payout-10001&signature=5ce1...
```

## 7. Практические рекомендации

- Валидируйте реквизиты клиента на своей стороне ещё до вызова API.
- После `S10002` сначала ищите payout по `externalOrderId`, а не создавайте новый.
- Не считайте `201 Created` финальным подтверждением успешной выплаты.
- Отдельно логируйте `status`, `statusDetails`, `externalOrderId` и `callbackUrlStatus`.
- Для проблемных кейсов сразу дочитывайте ордер по `GET`, а не полагайтесь только на UI.
