# Документация simple-pay

`simple-pay` предоставляет merchant API для `payin` и `payout`, callback-уведомления, работу с балансами магазина и набор служебных endpoint-ов для интеграции.

Эта документация написана так, чтобы интегратор мог стартовать, имея только:

- домен инстанса `simple-pay`;
- `Shop API key`;
- `Balance API key`;
- `Signature key` для callback.

## С чего начать, если у вас только домен и ключи

### 1. Соберите `BASE_URL`

Если у вас есть домен инстанса, merchant API обычно вызывается по префиксу:

```bash
export DOMAIN="[[DOMAIN_URL]]"
export BASE_URL="$DOMAIN/public/api/v1"
```

Дальше все merchant-запросы из документации используют именно `BASE_URL`.

### 2. Проверьте, что `Shop API key` работает

```bash
export SHOP_TOKEN="<SHOP_API_KEY>"

curl --location "$BASE_URL/shop/info" \
  --header "Authorization: Bearer $SHOP_TOKEN"
```

Если ключ корректный, вы получите базовую информацию о магазине.

### 3. Проверьте доступные методы

Для `payin`:

```bash
curl --location "$BASE_URL/shop/trade-methods" \
  --header "Authorization: Bearer $SHOP_TOKEN"
```

Для `payout`:

```bash
curl --location "$BASE_URL/shop/trade-methods/payout" \
  --header "Authorization: Bearer $SHOP_TOKEN"
```

### 4. Создайте первый тестовый ордер

- для приёма платежей начните со страницы [PAYIN: способы и API-примеры](/doc/payin/02-integration/)
- для выплат начните со страницы [PAYOUT: способы и API-примеры](/doc/payout/02-integration/)

### 5. Настройте получение статусов

Рекомендуемая схема:

- `callbackUrl` , основной канал статусов;
- `GET`-чтение ордера по `id` или `externalOrderId` , резервный механизм контроля и восстановления состояния.

С учётом того, что у интегратора есть `Signature key`, callback стоит реализовывать сразу как обязательную часть интеграции.

## Что вы можете делать через API

### PayIn

- создавать ордера;
- читать ордера по `id` и `externalOrderId`;
- обновлять payment-данные;
- запускать поиск реквизитов;
- подтверждать оплату;
- отменять ордер;
- открывать и закрывать dispute;
- загружать и читать чеки.

### PayOut

- получать доступные payout methods;
- создавать payout-ордера;
- читать payout по `id` и `externalOrderId`;
- отменять payout-ордеры в допустимых статусах;
- получать статусы через callback или `GET`.

### Shop / Service

- проверять магазин и курсы;
- читать баланс;
- создавать заявки на вывод средств магазина;
- читать справочники.

## Что важно понимать про “изменение статусов”

Merchant не записывает статус ордера произвольным полем вроде `status=completed`.

Статусы меняются через action-методы:

- `POST /shop/orders/{id}/start-payment`
- `POST /shop/orders/{id}/confirm-payment`
- `POST /shop/orders/{id}/cancel`
- `POST /shop/orders/{id}/dispute`
- `POST /shop/orders/{id}/dispute/cancel`
- `POST /shop/payout-orders/{id}/cancel`

То есть мерчант управляет не самим значением `status`, а бизнес-действиями, которые переводят ордер в следующий допустимый статус.
