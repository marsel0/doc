---
title: "Ошибки и коды ответов"
---

Эта страница описывает, как `simple-pay` возвращает ошибки, какие бизнес-коды используются в `payin` и `payout`, и как правильно реагировать на них в интеграции.

## 1. Два уровня ошибок

В API есть два основных типа ошибок:

### Ошибки валидации

Возникают, если payload не соответствует схеме запроса.

Пример:

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": [
    {
      "property": "amount",
      "constraints": {
        "isPositive": "amount must be a positive number"
      }
    }
  ]
}
```

Такие ошибки нужно исправлять в коде интеграции. Ретраить их бессмысленно.

### Бизнес-ошибки

Возникают, когда запрос структурно корректен, но не может быть обработан в текущем состоянии.

Типовой формат:

```json
{
  "error": "Bad Request",
  "errorCode": "S10000",
  "errorMessage": "Shop is inactive",
  "statusCode": 400
}
```

Для маршрутизации в коде ориентируйтесь на `errorCode`.

## 2. Типовые HTTP-ошибки

### 401 Unauthorized

Когда возникает:

- не передан `Authorization`;
- передан неверный ключ;
- использован не тот тип ключа;
- не пройдена IP-проверка магазина.

Пример:

```json
{
  "statusCode": 401,
  "message": "Unauthorized"
}
```

Что делать:

- проверить `Bearer`-токен;
- убедиться, что для `/shop/assets` используется `Balance API key`;
- проверить ограничения по IP.

### 404 Not Found

Когда возникает:

- ордер не найден;
- payout-ордер не найден;
- запрошен ресурс не из того магазина или окружения.

Пример:

```json
{
  "statusCode": 404,
  "message": "Order not found",
  "error": "Not Found"
}
```

Что делать:

- проверить `id` или `externalOrderId`;
- проверить, что вы обращаетесь в нужную среду;
- убедиться, что используете ключ именно того магазина, который создал ордер.

### 500 Internal Server Error

Когда возникает:

- внутренняя ошибка платформы;
- нестабильная внешняя зависимость;
- редкий edge case в обработке ордера.

Пример:

```json
{
  "statusCode": 500,
  "message": "Internal server error"
}
```

Что делать:

1. не создавать новый ордер автоматически;
2. сначала дочитать текущее состояние через `GET`;
3. если это create-запрос и передавался `externalOrderId`, искать ордер по нему;
4. если состояние осталось неясным, эскалировать кейс в поддержку.

## 3. Отдельно про `S10002` и таймауты ответа

`S10002` означает, что платформа не успела вернуть ответ в пределах таймаута магазина.

Это важная ошибка, потому что она не означает, что операция точно не произошла.

Пример:

```json
{
  "error": "Bad Request",
  "errorCode": "S10002",
  "errorMessage": "Response timeout was exceeded according to shop settings",
  "statusCode": 400
}
```

Что делать правильно:

1. найти ордер по `externalOrderId`;
2. если ордер найден, продолжать работу с ним;
3. только если ордер действительно не создан, принимать решение о повторе.

Это особенно важно для:

- `POST /shop/orders`
- `POST /shop/orders/sync-requisites`
- `POST /shop/payout-orders`

## 4. Основные бизнес-коды API

### Ошибки магазина

| Код | Что означает | Что делать |
| --- | --- | --- |
| `S10000` | магазин неактивен | остановить интеграцию и проверить статус магазина |
| `S10001` | недостаточно средств на балансе магазина | проверить `/shop/assets`, пополнить баланс или сократить объём выплат |
| `S10002` | превышен таймаут ответа магазина | дочитать ордер по `externalOrderId`, не дублировать create вслепую |

### Ошибки ордера

| Код | Что означает | Что делать |
| --- | --- | --- |
| `O10000` | действие запрещено в текущем статусе | сначала прочитать ордер и убедиться, что статус подходит для действия |
| `O10001` | не выбран `payment.type` и/или `bank` для старта payin flow | перед `start-payment` или `confirm-payment` заполнить payment-данные |
| `O10002` | превышен лимит активных ордеров на клиента | не создавать новые ордера в цикле, разбирать лимиты с проектом |
| `O10003` | ордер не найден | проверить `id`, `externalOrderId`, магазин и среду |
| `O10004` | некорректная сумма ордера | проверить лимиты магазина и диапазоны сценария |
| `O10005` | реквизиты не найдены | предложить другой `payment.type` или другой `bank` |
| `O10006` | такой `externalOrderId` уже существует | использовать уникальный внешний ID и искать существующий ордер |
| `O10007` | найдено больше одного ордера с тем же `externalOrderId` | эскалировать кейс в поддержку |
| `O10008` | клиент заблокирован антифродом | не ретраить бесконечно, предложить повтор позже |
| `O10009` | клиент не в whitelist | проверить настройки trusted customers |
| `O10010` | сумма запрещена антифродом | предложить другой номинал или пересмотреть лимиты |

### Ошибки валют, банков и методов

| Код | Что означает | Что делать |
| --- | --- | --- |
| `C10000` | не найден курс обмена | проверить валюту магазина и настройки exchange |
| `C10001` | валюта не найдена | проверить `currency` в payload |
| `B10000` | банк не найден | проверить `payment.bank` |
| `P10000` | тип оплаты не найден | проверить `payment.type` |
| `T10000` | trade method не найден | дочитать `GET /shop/trade-methods` или `GET /shop/trade-methods/payout` |

## 5. Частые ошибки в `payin`

### `O10005` на `POST /shop/orders/sync-requisites`

Это не просто абстрактная ошибка. В этом сценарии API обычно возвращает ещё и снимок созданного ордера.

Пример:

```json
{
  "statusCode": 404,
  "errorMessage": "No requisites found for specific payment type and bank at this moment",
  "errorCode": "O10005",
  "order": {
    "id": "1d481bac-61e0-4e9b-8b54-2c48d474d394",
    "amount": 132,
    "currency": "RUB",
    "status": "cancelled",
    "statusDetails": "requisites_timeout",
    "requisites": null
  }
}
```

Что делать:

- зафиксировать `order.id` и `externalOrderId`;
- предложить другой банк или другой `payment.type`;
- не повторять тот же запрос бесконечно с тем же методом.

### `O10000` на `confirm-payment`

Обычно возникает, если ордер ещё не перешёл в `customer_confirm` или уже финализирован.

Правильная реакция:

1. прочитать ордер по `GET`;
2. убедиться в `status`;
3. только потом повторять действие или завершать сценарий.

## 6. Частые ошибки в `payout`

### `S10001` при создании payout

Недостаточно баланса магазина.

Что делать:

- вызвать `GET /shop/assets`;
- проверить доступный баланс;
- не повторять `POST /shop/payout-orders`, пока баланс не исправлен.

### `O10000` при отмене payout

Возникает, если ордер уже ушёл в статус, в котором отмена недоступна.

Что делать:

- дочитать payout через `GET /shop/payout-orders/{id}`;
- завершить кейс по фактическому статусу, а не по ожиданию UI.

## 7. Стратегия ретраев

### Можно повторять после дополнительной проверки

- `S10002` , но только после поиска ордера по `externalOrderId`
- `500 Internal Server Error` , но только после повторного чтения ресурса

### Можно повторять позже

- `O10005` , если вы пробуете другой банк или другой способ оплаты
- `O10008` , если пользователю предложено повторить попытку позже

### Нельзя ретраить без изменения данных

- `401`
- ошибки валидации `400`
- `O10004`
- `O10006`
- `B10000`
- `P10000`
- `T10000`

## 8. Callback и ошибки доставки

Даже если сам ордер обработан корректно, доставка callback может не удаться.

Сигналы, которые стоит использовать:

- `integration.callbackUrlStatus = in_progress`
- `integration.callbackUrlStatus = success`
- `integration.callbackUrlStatus = error`

Что делать:

- логировать callback c `id`, `externalOrderId`, `status`, `statusDetails`;
- обрабатывать callback идемпотентно;
- если `callbackUrlStatus = error`, дочитывать ордер через `GET`.

В коде проекта callback отправляется повторно автоматически с backoff, поэтому одно и то же уведомление может прийти несколько раз.

## 9. Практический алгоритм обработки ошибок

### Для create-методов

1. Всегда передавайте `externalOrderId`.
2. Если пришёл `S10002` или `500`, ищите ордер по `externalOrderId`.
3. Только если ордер не найден, решайте, делать ли повторный create.

### Для action-методов

1. При `O10000` сначала дочитайте ордер.
2. Дальше принимайте решение по фактическому `status`.

### Для callback

1. Проверяйте `signature`.
2. Сохраняйте событие независимо от того, финальный статус или нет.
3. При конфликте между callback и вашей локальной БД доверяйте повторному `GET`.
