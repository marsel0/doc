title: "Антифрод"
--- 

Эта страница описывает флоу работы с антифродом со стороны мерчанта. Внутренние правила скоринга не раскрываются, но для корректной интеграции важны входные данные и реакция на ответы API.

## 1. Какие данные нужно передавать

Для устойчивой антифрод-оценки передавайте максимально стабильные данные клиента:

- `customer.id`
- `customer.ip`
- `customer.phone`
- `customer.email`
- `customer.fingerprint`
- сумму, валюту и выбранный `payment.type`

Если эти поля меняются случайно между одинаковыми попытками, система хуже сопоставляет клиента и может чаще возвращать ограничения.

## 2. Как встроить антифрод в основной флоу

1. Клиент выбирает способ оплаты или выплаты.
2. Вы создаёте ордер стандартным методом.
3. Если ордер создан успешно, продолжаете обычный flow.
4. Если API вернул антифрод-ошибку, останавливаете сценарий и не запускаете бесконечные ретраи.
5. Если антифрод включён в режиме whitelist, используйте только клиентов из заранее согласованного списка.

## 3. Как реагировать на ответы

- `O10008` — клиент заблокирован. Попросите клиента повторить попытку позже или сменить сценарий оплаты.
- `O10009` — клиент не в whitelist. Проверьте настройки проекта.
- `O10010` — сумма запрещена. Предложите другой диапазон суммы.

Подробный список кодов находится на странице [Ошибки и коды ответов](/doc/docs/02-api_error_guide/).

## 4. Практические рекомендации

- Не меняйте `customer.id` между повторами одного и того же бизнес-заказа.
- Не создавайте десятки ордеров подряд после отказа антифрода.
- Передавайте реальный IP клиента, а не IP вашего backend.
- Если используете device fingerprint, передавайте его стабильно во все повторные попытки.
- Для спорных кейсов логируйте `id`, `externalOrderId`, `customer.id`, `status`, `statusDetails`.
